# PromptMaker シリーズ ― 4フェーズ・ユーザージャーニー設計

`DESIGN.md` がコードの構造（レイヤー・ファイル構成）の設計図だとすると、
こちらは「人がこのツールをどんな順番で使うか」というUX側の設計図。
**まだ実装しない。設計を固めるためのメモ。**

## 全体像

| # | フェーズ | 目的 | 主に使うツール | 実装状況 |
|---|---|---|---|---|
| 1 | コンセプト決め | 何を・誰に・なぜ、を固める | LP作成（①基本設定） | 既存の項目がそのまま使える |
| 2 | アイコン・デザインの雰囲気 | フェーズ1を見た目に翻訳する | アイコン作成／画像作成／マスコット・相棒キャラ作成 | 画像作成のみ実装済み |
| 3 | メニュー・店舗情報・リンク | 実際に載せる中身を埋める | LP作成（掲載情報） | メニューのみ実装済み、リンクは未実装 |
| 4 | 詳細・仕上げ | 微調整して完成させる | LP作成（詳細設定・出力） | 既存の項目がそのまま使える |

軸になる考え方：**フェーズ1で決めた「トーン」が、フェーズ2以降ずっと効き続ける。** 各フェーズは独立した入力画面ではなく、フェーズ1の答えを前提に選択肢や既定値が変わっていくのが理想形。

---

## インタラクション形式：質問ゲート型ウィザード

フェーズ2は候補が3つ（画像作成／アイコン作成／マスコット作成）に増えたため、全部を固定ページとして並べるとページ数が膨らむ。
代わりに、**各サブツールの手前に「やりますか？」の一問一答（はい／いいえ）を置き、「はい」と答えたときだけその先の詳細ステップに進む**、という質問ゲート型にする。

例（フェーズ2部分）：

```
Q. 画像（写真素材）を作りますか？　［はい］［いいえ］
   → はい：画像作成の詳細ステップへ　／　いいえ：次の質問へスキップ

Q. アイコンを作りますか？　［はい］［いいえ］
   → はい：アイコン作成の詳細ステップへ　／　いいえ：次の質問へスキップ

Q. マスコットキャラクターを作りますか？　［はい］［いいえ］
   → はい：相棒キャラ作成の詳細ステップへ　／　いいえ：次のフェーズへ
```

同じ形はフェーズ3・4の「任意項目」にも応用できる（例：「リンクを載せますか？」「詳細をこだわりたいですか？」）。

**この形式にすると、ページ数は固定ではなく可変になる。**

| ケース | ページ数 |
|---|---|
| 全部「いいえ」（最小） | プリセット→①コンセプト→③中身→④完成 の **4ページ** |
| 画像作成だけ「はい」（今すぐ作れる範囲） | 上記＋画像作成1ページで **5〜6ページ** |
| 全部「はい」（アイコン・マスコットも実装済みの前提） | 最大 **8ページ** |

質問ゲート自体は1画面1問（「はい／いいえ」の2択チップのようなもの）にするか、まとめて1画面に並べるかはまだ未決定（下記の未確定事項に追加）。

## フェーズ1：コンセプト決め

**目的**：このLP/ブランドが何を・誰に・なぜ提供するかを1つに決める。以降すべてのフェーズの土台。

| 項目 | 現状 |
|---|---|
| 業種（type） | 既存（LP作成・basic） |
| 読み手（reader） | 既存（LP作成・basic） |
| ゴール（goal） | 既存（LP作成・basic） |
| トーン（tone） | 既存（LP作成・basic）※フェーズ2・3に伝播させたい値 |
| ボリューム（volume） | 既存（LP作成・basic） |

**出力**：「業種×読み手×ゴール×トーン」の組み合わせそのもの（`state` として保持）。

## フェーズ2：アイコン・デザインの雰囲気

**目的**：フェーズ1のコンセプトを「見た目」に翻訳する。

| ツール | 想定する入力 | 実装状況 |
|---|---|---|
| 画像作成 | 用途・被写体・比率・テイスト・ブランドカラー | 実装済み（`template.visual.js`）。AIが作った画像を戻して<br>セットする「画像を用意する」ステップも対応済み（下記） |
| アイコン作成 | 用途・モチーフ・スタイル（フラット/3D/ラインアート等）・カラー数・テイスト | 実装済み（`template.icon.js`）。同じく「画像を用意する」ステップに対応済み |

**画像を作った後の受け皿**：画像作成・アイコン作成の両方に `imageSlots` を追加したので、プロンプトをコピー→画像生成AIで作成→
できた画像をここに戻してセット、までLP作成と同じ流れでできる。セットすると自動でWebPに変換され、パターンごとの
スロットid（`pattern-1` / `icon-1`など）でファイル名が付き、まとめてZIPで書き出せる。これはLP作成専用の仕組みではなく
「`imageSlots(state)` を実装したテンプレートなら誰でも使える」共通機能（`DESIGN.md` §4参照）なので、
今後マスコット作成を実装するときも、同じ形でそのまま使い回せる。

**フェーズ1との連携**：画像作成・アイコン作成の「テイスト」はどちらもLP作成と同じ語彙（高級感・親しみやすい・ポップ・元気・信頼誠実・おしゃれ洗練・エモい）に揃えてある。タブ切替時に `tone` / `ref` / `extra` / `brandName` / `brandColor` は自動で引き継がれる（`app.js` の `carryStateAcrossTemplates()`）ので、フェーズ1のトーン1つで見た目の一貫性が保てる。

### フェーズ2の3つ目の選択肢：マスコット・相棒キャラ作成（相棒maker）

もちすら氏の別リポジトリ `mochipj2025/Promptmaker002` の `dev/companion-maker.html` に試作あり（未実装・移植候補）。
「アイコン作成」（favicon/アプリアイコンのような機能的な印）とは別物で、**ブランドの小さなマスコットキャラクター**を作る位置づけ。

- 元の仕様：ベース・モチーフ・色・目・チーク・口・アイテム・アクセント・性格の9カテゴリからランダムに1つずつ選び、キャラ説明文＋固定の「三面図」プロンプトを連結する。FRONT/SIDE/BACKの枠は見た目だけの点線プレースホルダーで、画像アップロード機能は無い（試作段階）。
- 移植する場合の設計：**基本の型（9カテゴリ＋三面図構成＋build()の組み立て方）は相棒makerをそのまま踏襲**しつつ、次の2点を拡張する。
  1. **ボタン（chips）式に改変**：9カテゴリを全部chipsフィールド化し、既存の「ランダム」ボタン（全項目シャッフル）はそのまま使えるうえで、1項目ずつ手動で選び直せるようにする（元は全部同時リロールしかできなかった）。名前は空欄なら自動生成、固定プロンプトは編集可能なテキストとして初期値に入れる。`imageSlots()` を実装すればFRONT/SIDE/BACKが本物の画像アップロード欄になる（元サイトより機能が上がる）。
  2. **「世界観」フィールドを新設**：かわいい系（元の相棒maker）に加えて、**サイバーパンク**・**スチームパンク**を選べるようにする。ベース/モチーフ等の9カテゴリの言葉はそのまま使い回し（例：モチーフの「ロボ」はサイバーパンクとも相性がいい）、**世界観ごとに「固定プロンプト」の文面を差し替える**ことで見た目のトーンをガラッと変える設計にする。
     - かわいい系（既定）：白に近い生成り背景・細い罫線・余白多め・「かわいいが騒がしくない」
     - サイバーパンク：ネオンの発色・金属パーツやケーブルを小物に効かせる・暗めの背景・「かわいいけれど無機質さも同居」
     - スチームパンク：真鍮・歯車・パイプなどアンティーク機械パーツ・セピア寄りの生成り背景・「骨董の図鑑のような落ち着いた設定資料風」
- **保留理由**：別リポジトリにはこの他にも `simple-character-promptmaker` `real-portrait-maker` `art-character-maker` `anime-comic-maker` `five-question-character-maker` というキャラクター系ツールが5本すでに存在する。相棒makerを足す前に、これらとの役割分担（何がどう違うのか）を整理してからにしたい。詳細は `DESIGN.md` の「外部エコシステムとの関係」を参照。

## フェーズ3：メニュー・店舗情報・リンク

**目的**：LPに実際に載せる具体的な情報を埋める。

| 項目 | 現状 |
|---|---|
| 店名・ブランド名（infoName） | 既存 |
| 商品・メニュー（menuItems） | 既存（＋ボタンで複数行対応済み） |
| 日時・期間（infoDate） | 既存 |
| 場所・アクセス（infoPlace） | 既存（テキストのみ。地図URLは持たない） |
| 特徴・こだわり（infoFeature） | 既存 |
| **リンク** | **未実装。種類は保留中** |

**リンクの設計方針（保留のまま進める場合の受け皿）**：
種類（SNS／予約／地図／ネットショップ等）を先に決め打ちせず、`menuItems` と同じ「repeater（行を＋で増減できる欄）」パターンで、

```
{ 種類: 自由入力 or 選択肢, URL: テキスト }
```

という汎用形にしておけば、種類の議論が先に進まなくても構造だけ先に用意できる。種類を選択式にしたくなったら、あとから options を追加するだけで済む。
