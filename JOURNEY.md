`DESIGN.md` がコードの構造（レイヤー・ファイル構成）の設計図だとすると、
こちらは「人がこのツールをどんな順番で使うか」というUX側の設計図。
**まだ実装しない。設計を固めるためのメモ。**

## 全体像

| # | フェーズ | 目的 | 主に使うツール | 実装状況 |
|---|---|---|---|---|
| 1 | コンセプト決め | 何を・誰に・なぜ、を固める | LP作成（①基本設定） | 既存の項目がそのまま使える |

## インタラクション形式：質問ゲート型ウィザード

フェーズ2は候補が3つ（画像作成／アイコン作成／マスコット作成）に増えたため、全部を固定ページとして並べるとページ数が膨らむ。
代わりに、**各サブツールの手前に「やりますか？」の一問一答（はい／いいえ）を置き、「はい」と答えたときだけその先の詳細ステップに進む**、という質問ゲート型にする。

例（フェーズ2部分）：

```
Q. 画像（写真素材）を作りますか？　［はい］［いいえ］
   → はい：画像作成の詳細ステップへ　／　いいえ：次の質問へスキップ

Q. アイコンを作りますか？　［はい］［いいえ］
   → はい：アイコン作成の詳細ステップへ　／　いいえ：次の質問へスキップ

Q. マスコットキャラクターを作りますか？　［はい］［いいえ］
   → はい：相棒キャラ作成の詳細ステップへ　／　いいえ：次のフェーズへ
```

同じ形はフェーズ3・4の「任意項目」にも応用できる（例：「リンクを載せますか？」「詳細をこだわりたいですか？」）。

**この形式にすると、ページ数は固定ではなく可変になる。**

| ケース | ページ数 |
|---|---|
| 全部「いいえ」（最小） | プリセット→①コンセプト→③中身→④完成 の **4ページ** |
| 画像作成だけ「はい」（今すぐ作れる範囲） | 上記＋画像作成1ページで **5〜6ページ** |
| 全部「はい」（v2がアイコン・マスコットのステップにも対応した場合） | 最大 **8ページ** |

質問ゲート自体は1画面1問（「はい／いいえ」の2択チップのようなもの）にするか、まとめて1画面に並べるかはまだ未決定（下記の未確定事項に追加）。

**現状（2026-07-18時点）**：画像作成・アイコン作成・マスコット作成はいずれも `index.html`（フル機能版）でタブとして実装済み。ただし v2（ステップ版）は依然 LP作成のみ対応で、この「質問ゲート型」で3つを繋ぐ統合フローそのものはまだ実装していない。

## フェーズ2：アイコン・デザインの雰囲気

**目的**：フェーズ1のコンセプトを「見た目」に翻訳する。

| ツール | 想定する入力 | 実装状況 |
|---|---|---|
| 画像作成 | 用途・被写体・比率・テイスト・ブランドカラー | 実装済み（`template.visual.js`）。AIが作った画像を戻して<br>セットする「画像を用意する」ステップも対応済み（下記） |
| アイコン作成 | 用途・モチーフ・スタイル（フラット/3D/ラインアート等）・カラー数・テイスト | 実装済み（`template.icon.js`）。同じく「画像を用意する」ステップに対応済み |
| マスコット作成 | かたち・モチーフ・いろ・め・ほっぺ・くち・アイテム・アクセント・せいかく（素材ボタンの組み合わせ）・世界観（かわいい／サイバーパンク／スチームパンク）・テイスト | 実装済み（`template.mascot.js`）。imageSlotsは三面図（正面／横／背面） |

**画像を作った後の受け皿**：画像作成・アイコン作成・マスコット作成の3つに `imageSlots` を追加したので、プロンプトをコピー→画像生成AIで作成→
できた画像をここに戻してセット、までLP作成と同じ流れでできる。セットすると自動でWebPに変換され、パターンごとの
スロットid（`pattern-1` / `icon-1` / `mascot-front`など）でファイル名が付き、まとめてZIPで書き出せる。これはLP作成専用の仕組みではなく
「`imageSlots(state)` を実装したテンプレートなら誰でも使える」共通機能（`DESIGN.md` §4参照）で、実際にマスコット作成でも
同じ仕組みをそのまま使い回せている（三面図の正面／横／背面が3スロットになる形）。

**フェーズ1との連携**：画像作成・アイコン作成・マスコット作成の「テイスト」はいずれもLP作成と同じ語彙（高級感・親しみやすい・ポップ・元気・信頼誠実・おしゃれ洗練・エモい）に揃えてある。タブ切替時に `tone` / `ref` / `extra` / `brandName` / `brandColor` は自動で引き継がれる（`app.js` の `carryStateAcrossTemplates()`）ので、フェーズ1のトーン1つで見た目の一貫性が保てる（`template.mascot.js` も同じキー名でfields定義しているだけで、特別な連携コードは書いていない）。

### フェーズ2の3つ目の選択肢：マスコット・相棒キャラ作成（相棒maker）― 実装済み（2026-07-18）

もちすら氏の別リポジトリ `mochipj2025/Promptmaker002` の `dev/companion-maker.html` の試作をヒントに、`template.mascot.js` として実装した。
「アイコン作成」（favicon/アプリアイコンのような機能的な印）とは別物で、**ブランドの小さなマスコットキャラクター**を作る位置づけ。

- 元の仕様（相棒maker試作）：ベース・モチーフ・色・目・チーク・口・アイテム・アクセント・性格の9カテゴリからランダムに1つずつ選び、キャラ説明文＋固定の「三面図」プロンプトを連結する。FRONT/SIDE/BACKの枠は見た目だけの点線プレースホルダーで、画像アップロード機能は無い（試作段階）。
- **実装した設計**：この計画どおり、**9カテゴリ（かたち・モチーフ・いろ・め・ほっぺ・くち・アイテム・アクセント・せいかく）を全部chipsボタン化**し、🎲ランダムで全項目シャッフル・1項目ずつの手動選び直しの両方に対応。`imageSlots()` を実装したのでFRONT/SIDE/BACK（正面／横／背面）は本物の画像アップロード欄になっている（元サイトより機能が上がった状態）。
- **「世界観」フィールドも計画通り新設**：かわいい（既定）・サイバーパンク・スチームパンクの3つで、世界観ごとに固定プロンプトの文面を差し替える設計をそのまま実装（`WORLD_FIXED` / `RENDER_STYLE`）。さらに画像生成AI（ChatGPT Image 2.0）向けに、プロンプトを「構図／画風／一貫性／背景／注意」の構造化フォーマットに強化し、文字ラベルの描き込みを明示的に禁止している。
- **LPとの連携も追加**：LP作成の掲載情報に「マスコット・相棒キャラ」欄を新設し、入力があるとLP制作プロンプト／設計図に登場指示（FV・区切り・CTA付近にさりげなく・画像パス指定）を差し込む形にした（未入力なら省略）。
- **住み分けの結論**：別リポジトリの `simple-character-promptmaker` `real-portrait-maker` `art-character-maker` `anime-comic-maker` `five-question-character-maker` はいずれも「画像生成AI向けの汎用キャラプロンプトメーカー」で、三面図対応も既にある。対して本実装は「**このLP/ブランド専用**・トーン継承・`imageSlots`によるLP素材ワークフロー連携」に絞ることで重複を避ける方針（＝薄く統合）で決着した。別リポとの詳細な役割分担・統合可否は `DESIGN.md` §7 参照（未解決のまま）。

## フェーズ3：メニュー・店舗情報・リンク

**目的**：LPに実際に載せる具体的な情報を埋める。

| 項目 | 現状 |
|---|---|
| 店名・ブランド名（infoName） | 既存 |
| 商品・メニュー（menuItems） | 既存（＋ボタンで複数行対応済み） |
| 日時・期間（infoDate） | 既存 |
| 場所・アクセス（infoPlace） | 既存 |
| Googleマップのリンク（infoMapUrl） | 実装済み（GEO/AEO対応。構造化データの`hasMap`・地図埋め込み指示に反映） |
| 特徴・こだわり（infoFeature） | 既存 |
| マスコット・相棒キャラ（mascot） | 実装済み。入力があるとLP出力に登場指示を追加 |
| **リンク（links）** | **実装済み**（種類は自由入力＋URLの汎用repeater。sameAsとして構造化データにも反映） |

**リンクの実装（2026-07-18）**：当初の方針どおり、種類（SNS／予約／地図／ネットショップ等）を決め打ちせず、`menuItems` と同じ「repeater（行を＋で増減できる欄）」パターンで

```
{ label: 自由入力, url: テキスト }
```

という汎用形で実装した。種類を選択式にしたくなったら、あとから options を追加するだけで済む（今のところ自由入力のまま）。入力されたURLはGEO/AEO対応の構造化データ`sameAs`にも自動で反映される。
